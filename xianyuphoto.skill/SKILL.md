# 生成闲鱼商品主图

根据真实商品参考图生成 3 张不同场景的闲鱼主图。优先保证商品真实性；不得为了画面美观改变商品本体或虚构商品信息。

## 工作流

按顺序执行以下步骤。

### 1. 检查输入

要求用户提供：

- 至少一张能清晰展示唯一商品主体的真实实拍图。
- 1–3 个核心卖点。

允许用户选填商品名称、品牌型号、新旧程度、瑕疵、配件、价格、目标买家和风格要求。

缺少实拍图或核心卖点时，先询问用户补充，不得开始生成。存在多个不同商品主体且无法确定目标商品时，先要求用户确认。

### 2. 识别商品信息

查看全部参考图，识别商品主体、用途、材质、结构和可能的使用环境。

- 未提供产品名称时，推断不含品牌的通用产品名称。
- 未提供目标买家时，根据实拍图和核心卖点推断。
- 不得根据外观、相似款、Logo 形状或上下文推测品牌。
- 只有用户明确提供品牌，或品牌文字能从图片中直接、清晰读取时，才使用品牌。
- 不得推断无法确认的型号、成色、规格、功能、配件或价格。

### 3. 匹配品类和场景词

读取 [references/product-category-scene-library.md](references/product-category-scene-library.md)。将商品匹配到一个主要类目，并按该类目中的顺序提取 3 组推荐场景词。

- 以图片主体为主要依据，产品名称和核心卖点为辅助依据。
- 多类目均可能时，选择与核心卖点和目标买家最匹配的一个类目。
- 信息不足以可靠判断时，询问用户确认。
- 类目场景词与通用场景冲突时，以类目场景词为准。

生成前向用户简要展示：通用产品名称、推断的目标买家、匹配类目和 3 组场景词。判断明确时继续执行，不要求额外确认；判断存在实质歧义时才暂停询问。

### 4. 组装三个独立提示词

读取 [references/prompt-template.md](references/prompt-template.md)。为 3 张图片分别创建完整提示词，不要只用一个提示词请求 3 个随机变体。

- 第 1 张使用第 1 组类目场景词，突出日常实用性。
- 第 2 张使用第 2 组类目场景词，突出耐用、方便或省心。
- 第 3 张使用第 3 组类目场景词，突出材质、结构、容量、做工或功能区域，同时保持商品完整可见。
- 默认比例使用 `3:4`；用户明确要求时使用 `1:1`。
- 三张图都属于主图候选方案。不得扩展生成卖点图、详情图或带文案图片。

### 5. 调用 Codex 原生 Image Gen

在 Codex 桌面或提供原生 `image_gen` 工具的运行环境中，直接调用 Codex 原生图片生成能力，不经过外部图片生成 Skill、CLI、API 或第三方提供方。

- 将用户原始实拍图作为每个任务的参考图直接传给原生图片生成工具；不得仅依据文字重建商品。
- 本地参考图在调用前先使用可用的图片查看能力加载，确保其进入当前会话的视觉上下文。
- 默认要求 `3:4` 竖图；用户明确指定时改为 `1:1`。在每个独立提示词中重复比例要求。
- 顺序执行 3 个独立生成任务，一次只生成一张图片。三个任务必须共用同一组原始参考图。
- 不得使用任何新生成图片作为后续任务的参考图，避免商品主体逐步漂移。
- 每次调用都重复商品身份一致性、手机实拍质感和禁止项约束，并加入对应的场景词与展示目的。
- 原生工具调用失败时报告具体错误，不要静默切换到其他 Skill、CLI、API 或图片提供方。
- 生成完成后将最终图片复制到下述项目输出目录；不得只保留在 Codex 默认生成目录中。

将结果保存到 `outputs/{商品主体}/`：

- `01-{商品主体}-{场景简称}.png`
- `02-{商品主体}-{场景简称}.png`
- `03-{商品主体}-细节展示.png`

清理 Windows 文件名不支持的字符；同名时追加递增序号，不得覆盖已有文件。

### 6. 自动质检

读取 [references/quality-checklist.md](references/quality-checklist.md)。使用可用的图片查看或视觉检查能力逐张对照原始参考图，并验证尺寸比例。

- 逐张给出通过或不通过结论。
- 不合格时列出图片编号、失败项和具体问题。
- 不得自动重新生成；先询问用户是重新生成、修改提示词后生成，还是保留结果。
- 只重新处理用户确认的不合格图片，不重复生成合格图片。

## 输出边界

- 只交付 3 张闲鱼商品主图候选方案及质检结果。
- 不添加文字、水印、贴纸、价格、促销标签、箭头或夸张光效。
- 不生成卖点图、详情图、商品标题、价格建议或发布文案。
